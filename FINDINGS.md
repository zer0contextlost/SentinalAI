# SentinalAI: AI Code Detection via Style Fingerprinting
## An Empirical Study — and a Cautionary Result

---

## Abstract

We investigated whether machine learning classifiers trained on stylometric features
can reliably detect AI-generated code. Using a 65-feature extractor (lexical + Python
AST features), a 500K-sample multi-model corpus (SemEval-2026 Task 13, 34 AI models),
and a novel paired Codeforces corpus (96 problems solved by both humans and AI), we
achieve 95–99% F1 within a single dataset — and 40–61% F1 across datasets. We conclude
that current style-based detectors are learning dataset-specific artifacts, not universal
AI authorship signals. High in-domain accuracy is not evidence of generalization.

---

## 1. Motivation

AI-generated code has characteristic style: verbose, defensively structured,
well-commented, with consistent naming conventions. The hypothesis was that these
patterns are strong and consistent enough to serve as a reliable authorship fingerprint —
that a classifier could learn "this looks like AI wrote it" independent of domain or
which model produced it.

We set out to test this with proper controlled experiments rather than assuming
high in-domain accuracy implies a generalizable detector.

---

## 2. Datasets

### 2.1 SemEval-2026 Task 13 Corpus
- **Source**: HuggingFace `DaniilOr/SemEval-2026-Task13`, subtask A
- **Size**: 499,999 samples — 238,474 human, 261,525 AI
- **AI generators**: 34 models including GPT variants, CodeLlama, Qwen-Coder,
  deepseek-coder, StarCoder, Phi-3, Llama-3, CodeGemma, Granite
- **Languages**: Python (91%), C++ (5%), Java (4%)
- **Human samples**: scraped from real codebases, mixed era

### 2.2 Paired Codeforces Corpus (novel)
- **Design**: Same competitive programming problem, two solutions — one human, one AI
- **Human solutions**: Python files from GitHub repos last committed before 2022-01-01,
  verified via per-file git commit history
- **AI solutions**: Generated by `deepseek-coder:6.7b` via Ollama
- **Problems**: 96 problems from Codeforces contests ≤ 1500 (pre-2021 era),
  difficulty 800–1400, tag: implementation
- **Size**: 373 solutions — 273 human, 100 AI
- **Rationale**: Controls for era, domain, and problem difficulty simultaneously.
  Any measured difference is attributable to authorship, not subject matter or time period.

---

## 3. Feature Extraction

A 65-feature extractor operating on raw source text, combining:

**Lexical features (37)**: character and line counts, line length statistics (mean,
max, p90), blank line ratio, comment density, whitespace ratio, trailing whitespace,
digit/uppercase ratios, snake_case/camelCase counts, type annotation density, error
handling patterns (bare except, uniform except), defensive coding density (null checks,
isinstance, guard clauses, early returns), logging calls.

**Python AST features (28)**: function/class count, max nesting depth, comprehension
count, lambda/decorator usage, docstring presence, type annotations, f-string usage,
walrus operator, import count, max function length, and defensive pattern counts at
the AST level (none comparisons, isinstance calls, early returns, broad except handlers,
finally blocks, default arguments).

No perplexity or embedding features were used in the final classifier (see Section 5).

## 4. Results

### 4.1 In-domain: SemEval Corpus

| Model | F1 Macro | Accuracy |
|---|---|---|
| Random Forest | **0.9544** | 0.9544 |
| Gradient Boosting | 0.9534 | 0.9534 |
| Logistic Regression | 0.8758 | 0.8758 |

*5-fold stratified CV, 100K sample.*

**Top feature**: `blank_line_ratio` (importance 0.1741). AI-generated code inserts
blank lines between logical blocks consistently; human code varies widely.

### 4.2 In-domain: Paired Codeforces Corpus

Leave-one-problem-out (LOPO) cross-validation — all solutions for a given problem
are held out entirely during training, preventing any leakage between paired solutions.

| Model | F1 Macro | Accuracy |
|---|---|---|
| Logistic Regression | **0.9966** | 0.9973 |
| Random Forest | 0.9931 | 0.9946 |
| Gradient Boosting | 0.9932 | 0.9946 |

**Top features**:
- `ast_max_nesting_depth` (0.2297) — AI writes flatter code; human solutions nest deeper
- `max_line_length` (0.1658) — AI generates longer lines consistently
- `avg_line_length` (0.1607)
- `digit_ratio` (0.0537)

### 4.3 Cross-dataset Generalization

The critical test. Train on one corpus, evaluate on the other.

| Train → Test | Best F1 | Notes |
|---|---|---|
| Paired CF → SemEval 34-model | 0.6089 | Partial; large-model instruct outputs missed |
| SemEval 34-model → Paired CF | **0.4038** | Worse than random chance |

**Per-model breakdown (Paired → SemEval)**: Detection accuracy varies from 3.6%
(Llama-3.3-70B-Instruct) to 63% (Yi-Coder-1.5B). Large instruction-tuned models
are essentially undetectable by a classifier trained on deepseek-coder output.

**The SemEval → Paired result (40% F1) is the most telling.** A model trained on
100,000 balanced samples from 34 AI generators performs worse than flipping a coin
on the paired ground-truth corpus. The features it learned are anti-correlated with
AI-ness in the controlled setting.

## 5. Perplexity Signal: A Null Result

Token-level perplexity via Ollama continuation scoring (deepseek-coder:6.7b, first
800 chars as context, 60 generated tokens scored) was tested on 200 human + 200 AI
Python samples from SemEval.

| Feature | Human mean | AI mean | p-value |
|---|---|---|---|
| perp_mean_logprob | -0.5665 | -0.5760 | 0.35 |
| perp_perplexity | 1.7817 | 1.8024 | 0.35 |
| perp_variance | 0.4845 | 0.4967 | 0.35 |
| perp_surprising_ratio | 0.0072 | 0.0061 | 0.24 |
| perp_confident_ratio | 0.6043 | 0.6015 | 0.56 |

**0/8 features significant (Mann-Whitney U, p < 0.05).**

The hypothesis — that AI code context produces more confident model continuations —
did not hold. Deepseek-coder is equally fluent continuing both human and AI Python.
This may reflect that modern code models are trained on mixed corpora and have no
strong "recognition" bias toward AI-originated code.

---

## 6. Analysis: What Are the Classifiers Actually Learning?

The cross-dataset collapse reveals the underlying problem.

**SemEval signal (blank_line_ratio, whitespace_ratio)**: Formatting artifacts. AI
models generate code with consistent blank-line insertion between blocks. Human code
from mixed sources does not. This is defeated by any code formatter and likely reflects
how the human training data was scraped, not something intrinsic to human authorship.

**Paired corpus signal (nesting_depth, line_length)**: Population-level style gap.
Competitive programmers on GitHub write terse, deeply-nested solutions. An
instruction-tuned model generates verbose, flat, well-structured code. These populations
are stylistically extreme opposites, which inflates accuracy — but it is not an "AI
fingerprint," it is a "competitive programmer vs. RLHF assistant" fingerprint.

The 40% cross-dataset result is the proof: when the human population changes (from
competitive programmers to whatever SemEval scraped), the classifier's learned
associations invert.

This is the classic **distribution shift problem in stylometry**, well-documented in
authorship attribution literature. The problem is that "AI code" is not a single
distribution — it is the union of outputs from hundreds of models trained with
different objectives, on different data, with different post-training procedures.
A classifier trained on any subset of this distribution will not generalize to
models it has not seen.

---

## 7. Conclusions

1. **Style-based AI code detectors achieve high in-domain accuracy but do not generalize.**
   The gap between 99.7% (in-domain, paired) and 40% (cross-domain) is not a tuning
   problem — it reflects that the features are detecting dataset provenance, not authorship.

2. **Blank line ratio and formatting features are artifacts, not signals.** The top
   feature across most experiments is `blank_line_ratio`, which reflects how training
   data was collected and how AI models format output by default. It is not a stable
   authorship fingerprint.

3. **Perplexity-based detection, as implemented, yields no signal.** Continuation
   scoring via a code model does not distinguish human from AI Python code at 200
   samples per class.

4. **In-domain detection may still be useful in constrained deployments.** A detector
   trained on a specific model's output, evaluated within a fixed domain (e.g., a
   single course's assignment corpus), may perform adequately — but this must be
   explicitly scoped and validated per deployment.

5. **High accuracy on a labeled dataset is not evidence of a working detector.**
   Without cross-domain and cross-model validation, reported accuracies in AI code
   detection are likely measuring dataset construction choices.

---

## 8. Limitations and Future Work

- **Single AI model in paired corpus**: All AI solutions were generated by one model
  (deepseek-coder:6.7b). Results may improve with a multi-model paired corpus.

- **Competitive programming is an extreme domain**: Human solutions from this domain
  are atypically terse. Results may differ for production code, homework assignments,
  or open-source contributions.

- **Features do not cover semantics**: Identifier naming patterns, comment semantics,
  and algorithmic choice were not captured. These may carry stronger cross-domain signal.

- **Formatter invariance untested**: Whether reported signals survive `black` or
  `clang-format` normalization was not measured.

## Reproducibility

All code, data collection scripts, and feature extraction pipelines are in this
repository. The SemEval corpus is available at `DaniilOr/SemEval-2026-Task13` on
HuggingFace. Paired corpus collection requires a GitHub API token.

```
corpus/processed/features.parquet        # 500K SemEval feature matrix
corpus/processed/paired_features.parquet # 373-row paired CF feature matrix
corpus/paired/codeforces/                # Raw paired solutions
features/extractor.py                    # 65-feature extractor
scripts/train_paired_classifier.py       # LOPO-CV training
scripts/test_generalization.py           # Cross-dataset evaluation
scripts/test_generalization_reverse.py   # Reverse cross-dataset evaluation
```
