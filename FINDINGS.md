# SentinalAI: AI Code Detection via Style Fingerprinting
## An Empirical Study — and a Cautionary Result

---

## Abstract

We investigated whether machine learning classifiers trained on stylometric features
can reliably detect AI-generated code. Using a 65-feature extractor (lexical + Python
AST features), a 500K-sample multi-model corpus (SemEval-2026 Task 13, 34 AI models),
a novel paired Codeforces corpus (96 problems, human vs. AI), and HumanEval (164
problems, OpenAI canonical solutions vs. deepseek-coder), we achieve 0.95–0.99 F1
within a single corpus — and near or below random chance across corpora.

The failure is not random. A feature overlap analysis across three corpora reveals
the mechanism: **AI distributions are stable across corpora; human distributions
are not.** Detectors trained on one corpus implicitly learn where that corpus's
human code sits in feature space, not where AI code sits. When the human baseline
shifts — scraped code to competitive programmers to expert-written canonical
functions — the decision boundary inverts. Training on a union of diverse human
populations recovers 0.95 F1 and confirms the diagnosis.

A practical diagnostic, the **Human Baseline Distance (HBD) scorer**, is provided
as a continuous score rather than a binary label, with AUROC ranging from 0.975
(competitive programming context) to 0.540 (expert-canonical context).

---

## 1. Motivation

AI-generated code has characteristic style: verbose, defensively structured,
well-commented, with consistent naming conventions. The hypothesis was that these
patterns are strong and consistent enough to serve as a reliable authorship fingerprint —
that a classifier could learn "this looks like AI wrote it" independent of domain or
which model produced it.

We set out to test this with proper controlled experiments rather than assuming
high in-domain accuracy implies a generalizable detector.

---

## 2. Datasets

### 2.1 SemEval-2026 Task 13 Corpus
- **Source**: HuggingFace `DaniilOr/SemEval-2026-Task13`, subtask A
- **Size**: 499,999 samples — 238,474 human, 261,525 AI
- **AI generators**: 34 models including GPT variants, CodeLlama, Qwen-Coder,
  deepseek-coder, StarCoder, Phi-3, Llama-3, CodeGemma, Granite
- **Languages**: Python (91%), C++ (5%), Java (4%)
- **Human samples**: scraped from real codebases, mixed era

### 2.2 Paired Codeforces Corpus (novel)
- **Design**: Same competitive programming problem, two solutions — one human, one AI
- **Human solutions**: Python files from GitHub repos last committed before 2022-01-01,
  verified via per-file git commit history
- **AI solutions**: Generated by `deepseek-coder:6.7b` via Ollama
- **Problems**: 96 problems from Codeforces contests <= 1500 (pre-2021 era),
  difficulty 800–1400, tag: implementation
- **Size**: 373 solutions — 273 human, 100 AI
- **Rationale**: Controls for era, domain, and problem difficulty simultaneously.
  Any measured difference is attributable to authorship, not subject matter or time period.

### 2.3 HumanEval Corpus (third-corpus validation)
- **Source**: HuggingFace `openai/openai_humaneval`
- **Size**: 328 samples — 164 human, 164 AI (perfectly balanced)
- **Human solutions**: OpenAI canonical Python function implementations — expert-written,
  idiomatic, heavily reviewed
- **AI solutions**: Generated by `deepseek-coder:6.7b` via Ollama using the problem prompt
- **Rationale**: Third unseen corpus for out-of-distribution validation; human baseline
  is expert-quality code rather than scraped or competitive submissions

---

## 3. Feature Extraction

A 65-feature extractor operating on raw source text, combining:

**Lexical features (37)**: character and line counts, line length statistics (mean,
max, p90), blank line ratio, comment density, whitespace ratio, trailing whitespace,
digit/uppercase ratios, snake_case/camelCase counts, type annotation density, error
handling patterns (bare except, uniform except), defensive coding density (null checks,
isinstance, guard clauses, early returns), logging calls.

**Python AST features (28)**: function/class count, max nesting depth, comprehension
count, lambda/decorator usage, docstring presence, type annotations, f-string usage,
walrus operator, import count, max function length, and defensive pattern counts at
the AST level (none comparisons, isinstance calls, early returns, broad except handlers,
finally blocks, default arguments).

No perplexity or embedding features were used in the final classifier (see Section 5).

## 4. Results

### 4.1 In-domain: SemEval Corpus

| Model | F1 Macro | Accuracy |
|---|---|---|
| Random Forest | **0.9544** | 0.9544 |
| Gradient Boosting | 0.9534 | 0.9534 |
| Logistic Regression | 0.8758 | 0.8758 |

*5-fold stratified CV, 100K sample.*

**Top feature**: `blank_line_ratio` (importance 0.1741). AI-generated code inserts
blank lines between logical blocks consistently; human code varies widely.

### 4.2 In-domain: Paired Codeforces Corpus

Leave-one-problem-out (LOPO) cross-validation — all solutions for a given problem
are held out entirely during training, preventing any leakage between paired solutions.

| Model | F1 Macro | Accuracy |
|---|---|---|
| Logistic Regression | **0.9966** | 0.9973 |
| Random Forest | 0.9931 | 0.9946 |
| Gradient Boosting | 0.9932 | 0.9946 |

**Top features**:
- `ast_max_nesting_depth` (0.2297) — AI writes flatter code; human solutions nest deeper
- `max_line_length` (0.1658) — AI generates longer lines consistently
- `avg_line_length` (0.1607)
- `digit_ratio` (0.0537)

### 4.3 In-domain: HumanEval Corpus

5-fold stratified CV on 328 balanced samples (all 65 features):

| F1 Macro | Accuracy |
|---|---|
| **0.9200** | 0.9200 |

Confirms: the signal exists locally in every corpus. The features carry information —
it just does not transfer.

### 4.4 Cross-dataset Generalization (the critical test)

Train on one corpus, evaluate on another. HumanEval was never seen during feature
selection or training in these experiments.

| Train | Test | F1 Macro | Notes |
|---|---|---|---|
| Paired CF | SemEval | 0.6089 | Partial; large instruct models missed |
| SemEval | Paired CF | **0.4038** | Worse than random chance |
| SemEval | HumanEval | 0.3320 | Worse than random chance |
| SemEval (stable only) | HumanEval | 0.4298 | +0.098 lift with 9 stable features |
| Codeforces | HumanEval | 0.4449 | Below chance |
| SemEval+CF | HumanEval | 0.3320 | Combined training does not help |

**Union training (SemEval + CF + HumanEval combined, 5-fold CV):**

| Corpus slice | F1 Macro |
|---|---|
| SemEval | 0.9612 |
| Codeforces | 0.7817 |
| HumanEval | 0.8101 |
| **Overall** | **0.9517** |

Diverse human baseline recovers 0.95 F1. This is the fix — and the diagnosis.

## 5. Perplexity Signal: A Null Result

Token-level perplexity via Ollama continuation scoring (deepseek-coder:6.7b, first
800 chars as context, 60 generated tokens scored) was tested on 200 human + 200 AI
Python samples from SemEval.

| Feature | Human mean | AI mean | p-value |
|---|---|---|---|
| perp_mean_logprob | -0.5665 | -0.5760 | 0.35 |
| perp_perplexity | 1.7817 | 1.8024 | 0.35 |
| perp_variance | 0.4845 | 0.4967 | 0.35 |
| perp_surprising_ratio | 0.0072 | 0.0061 | 0.24 |
| perp_confident_ratio | 0.6043 | 0.6015 | 0.56 |

**0/8 features significant (Mann-Whitney U, p < 0.05).** Not pursued further.

## 6. Pivot Analyses

### 6.1 Model Fingerprinting (35-way Classification)

Trained a Random Forest to distinguish 34 AI models + human from stylometric features
alone. 1,000 samples per class, 5-fold stratified CV.

- **Macro F1 = 0.2342** vs. chance = 0.0286 (~8x above chance)
- **Human class** is the most distinguishable (F1 = 0.622)
- Family clusters match architecture lineage: Phi-3 variants cluster, Yi-Coder variants
  cluster, Llama derivatives group together
- Most confused pair: `starcoder2-15b` <-> `starcoder2-3b` (23% confusion rate)

Fine-tuning preserves base-model stylometric signatures. Within-corpus, AI models have
distinguishable fingerprints — they just don't transfer out of corpus.

### 6.2 Feature Forensics (Distribution Shift Analysis)

Measured Wasserstein distance and directional shift for each feature between SemEval
and Codeforces distributions.

**Conflict features** (significant in both corpora, opposite directions):

| Feature | SemEval delta (H->AI) | CF delta (H->AI) |
|---|---|---|
| line_count | +17.1 (AI writes more) | -10.9 (AI writes less) |
| non_blank_line_count | +12.1 | -10.8 |
| comment_line_ratio | positive | negative |
| whitespace_ratio | positive | negative |
| print_guard_count | positive | negative |
| camel_case_count | positive | negative |
| inline_comment_ratio | positive | negative |

**Stable features** (same direction, both corpora significant):
`max_line_length`, `avg_line_length`, `digit_ratio`, `p90_line_length`,
`naming_style_ratio`, `early_return_count`, `snake_case_count`,
`trailing_ws_ratio`, `long_ident_count`

Using only the 9 stable features: cross-domain F1 **0.38 -> 0.62** on SemEval->CF.

## 7. The Mechanism: Human Distributions Drift, AI Distributions Don't

The cross-corpus failures have a specific cause. For the top-10 discriminative features
(ANOVA on SemEval), HumanEval canonical humans sit inside SemEval-AI territory on
**10/10 features**:

| Feature | SemEval-H | SemEval-AI | HumanEval-H | HumanEval-AI |
|---|---|---|---|---|
| blank_line_ratio | 0.03 | **0.16** | **0.15** | 0.14 |
| char_count | 442 | **1027** | **631** | 677 |
| snake_case_count | 1.42 | **5.60** | **4.18** | 4.73 |
| line_count | 21.95 | **39.08** | **20.49** | 20.96 |
| naming_style_ratio | 0.15 | **0.45** | **0.63** | 0.68 |
| avg_line_length | 20.5 | **26.4** | **29.4** | 30.8 |

Bold = where the value sits relative to the SemEval boundary.

**The SemEval classifier has no choice but to call HumanEval humans "AI."** It learned
the position of SemEval's scraped human code in feature space. HumanEval's
expert-written canonical functions sit in the AI zone on every discriminative feature.

This is not domain shift in the generic sense. It is specifically **human-quality drift**:
when humans write exemplary, polished, expert-reviewed code, their surface style
converges toward AI. The detector was measuring "code quality" or "polish," not
AI provenance.

## 8. Human Baseline Distance (HBD) Scorer

Given that binary AI/human classification fails cross-corpus, we provide a continuous
diagnostic: the **HBD score**, measuring how far a code sample sits from the union
human distribution on the 9 stable features.

**Design:**
- Fit Mahalanobis distance on union human baseline (SemEval + CF + HumanEval, ~5,400 humans)
- Covariance estimated with Ledoit-Wolf shrinkage (condition number: 1.2 — well-conditioned)
- Features standardized on union, not per-corpus
- Output: distance in standard deviations from the diverse human mean
- **No threshold, no binary label** — diagnostic only

**AUROC (how well HBD separates human from AI within each corpus):**

| Corpus | Mean HBD (human) | Mean HBD (AI) | AUROC |
|---|---|---|---|
| SemEval | 1.78 | 4.61 | **0.798** |
| Codeforces | 1.83 | 8.06 | **0.975** |
| HumanEval | 4.35 | 4.56 | **0.540** |

The HumanEval AUROC of 0.540 is not a failure of the scorer — it is the thesis
quantified. Expert-written canonical humans (HBD 4.35) are indistinguishable from
AI (HBD 4.56) on the stable features, because they are both far from the "average"
human in the union distribution. The scorer is honest about what it cannot know.

**Practical interpretation:**
- High HBD (>6): code is anomalous relative to diverse human writing — worth human review
- Low HBD (<2): code sits well within the human distribution
- Moderate HBD (2–6): ambiguous — depends heavily on who the author population is

The confidence of the inference scales with how well your human population is
represented in the training baseline. A CF-context deployment (AUROC 0.975) is
actionable. A HumanEval-context deployment (AUROC 0.540) is not.

---

## 9. Conclusions

1. **Style-based detectors learn human baselines, not AI signatures.** The dominant
   source of variance across corpora is the human distribution, not the AI distribution.
   Reported F1 numbers measure corpus identity, not AI provenance.

2. **Human-quality drift is the failure mechanism.** When humans write expert-quality,
   polished code (HumanEval), they look like AI on every discriminative feature.
   The detector conflates "code quality" with "AI authorship."

3. **The signal is local and real.** Within-corpus F1 is high for all three corpora
   (0.92–0.99). The features carry information; it just does not transfer without
   a representative human baseline.

4. **Diverse training fixes it.** Union training on three human populations recovers
   0.95 F1. The practical recommendation is corpus diversity, not feature engineering.

5. **A continuous HBD score is more honest than a binary label.** Given the baseline-
   dependence of the signal, a distance score with explicit calibration bounds is more
   deployable than a classifier that silently fails when the human population shifts.

6. **Per-model fingerprinting works in-domain** (~8x above chance, 35-way classification),
   suggesting model-specific detectors trained and deployed within a controlled corpus
   may still be useful in constrained contexts (e.g., a single course, a single codebase).

7. **High in-domain accuracy is not evidence of generalization.** This cannot be stated
   enough: the gap between 99.7% (paired, in-domain) and 40% (cross-corpus) is not a
   tuning problem.

---

## 10. Limitations

- **Single AI model in paired corpora**: Codeforces and HumanEval AI solutions are both
  deepseek-coder:6.7b. A multi-model paired corpus may shift results.
- **Competitive programming is an extreme domain**: Human solutions are atypically terse.
- **Features do not cover semantics**: Identifier naming semantics, algorithmic choice,
  and comment content were not captured.
- **Formatter invariance untested**: Whether signals survive `black` normalization was
  not measured.
- **HBD calibration is union-dependent**: The scorer's confidence bounds apply to the
  specific union human population used here; deployment in new contexts requires
  recalibration against a local human baseline.

repository. The SemEval corpus is available at `DaniilOr/SemEval-2026-Task13` on

requires a GitHub API token.
